《你不知道的JavaScript》读书笔记（八）----Promise

  • 一旦Promise决议，它就永远保持在这个状态。此时它就成为了不变值

  • 两种Promise的实现方式：

    
  function bar(fooPromise) {
  	// 侦听foo(...)的完成
  	fooPromise.then(
  		function(){
  			// `foo(..)` 已经完成，所以开始做 `bar(..)`的任务
  		},
  		function(){
  			// 啊,  `foo(..)`出错了
  		}
  	);
  }
  
    
  function bar() {
  	// `foo(..)` 肯定已经完成了，所以执行 `bar(..)`的任务
  }

    function oopsBar() {
    
// 啊,  foo(..)出错了，所以bar(..) 没有执行
    
}
    

    // 对于 baz() 和 oopsBaz()同理
    

    var p = foo( 42 );
    

    p.then( bar, oopsBar );
    

    p.then( baz, oopsBaz );



    两种方法的区别是第一种方法bar(...)一定会被执行，并且会亲自处理错误的情况

  • 知道某个值是否是Promise很重要！！

    • 但是instance这个方法并不管用
    • 实际方法是判断其是否有then方法
    • 这种判断其具有那些属性，对值的类型做出判断。一般将其叫为鸭子类型

  • Promise的信任问题：

    • 调用过早对于一个Promise调用then的时候，即使这个Promise已经决议，提供给then的回调也总会被异步调用
    • 调用过晚一个Promise决议后，这个Promise上所有的通过then(...)注册的回调都会在下一个异步时机点上立即调用
    • 回调未调用没有任何东西，可以阻止Promise向你通知它的决议，即使如果Promise可能本身永远不会决议，也有相应的超时机制
    • 未能传递参数若没有用任何值显示决议，那么这个值就是undefined；JavaScript中的函数总是保持其定义所在的作用域的闭包
    • 吞掉错误Promise创建过程中的错误，这个错误会被捕捉，并使这个Promise被拒绝(Promise甚至把JS异常也变成了异常行为)；在then(...)注册的回调上出现错误，会使then(...)调用本身返回的Promise被拒绝
    • **“是可信任的Promise吗”：**使用Promise.resolve确保返回的是一个可信任的Promise
      • 如果向Promise.resolve(...)传递一个非Promise,非thenable的立即值，就会得到一个用这个值填充的Promise
      • 如果向Promise.resolve(...)传递一个真正的Promise，就只会返回同一个Promise

  • 链式流

    • 能实现链式的两个原因：
      • then(...)会返回一个Promise
      • 不管then(...)回调返回的值是什么，它都会被自动设置为被连接的Promise的完成(resolve)
    • 当你不传入拒绝处理函数，会有一个默认的拒绝处理函数，它只是把错误重新抛出，这使得错误可以沿着Promise传播下去，直到遇到显示定义的拒绝处理函数
    • then( null , function(err) {...} )只处理拒绝，相当于catch( function(err) {...} )
    • reject(...)不会像resolve(...)一样进行展开，如果向reject(...)传入一个Promise/thenable值，它会把这个值原封不动地设置为拒绝理由

  • 错误处理

    • try...catch(...)无法执行异步操作，node采用error-first的回调风格，Promise采用 分离回调风格
    • Promise的错误处理是“绝望的陷阱”，你必须努力才能做对，任何Promise链的最后一步，不管是什么，总是存在未被查看的Promise中出现未被捕获的错误
    • 成功的坑 ？....没看懂啊

  • Promise模式

    • Promise.all( [p1,p2] )，类似于“门”的机制，要等待两个或更多并行/并发的任务都完成才能继续。它们完成的顺序并不重要，但是必须都要完成，门才能打开并让流程控制继续。从Promise.all返回的promise会受到一个完成消息msg，它是传入的promise的完成消息组成的数组，与指定的顺序一致
    • Promise.race([...])，一旦有任何一个Promise决议为完成Promise.race([...])就会完成，一旦有任何一个Promise决议为拒绝，Promise.race([...])就会拒绝
      • 不过注意，如果传入一个空数组，则Promise永远不会决议
    • 变体：
      • none([...])全都被拒绝，决议为完成值
      • any([...])会忽略拒绝，只需完成一个就被决议为完成
      • firsst([...])只要第一个Promise完成，就会忽略后续的任何拒绝与完成
      • last([...])类似first，但却只有最后一个完成胜出

  • Promise API

    • new Promise(...)构造器
    • Promise.resolve(...)和promise.reject(...)，其中Promise.resolve( function(resolve,reject) )，就相当于new Promise ( function(resolve,reject) )
    • then(...)和catch(...),前者可注册完成和拒绝处理函数，后者只用来处理拒绝函数
    • Promise.all( [p1,p2] )，从MDN的描述看来当有任意一个拒绝，则所有的Promises立即失败，并且丢弃其他promises

  • Promise的局限性

    • 顺序错误处理：Promise链中的错误很容易被无意中忽略掉
    • 单一值：分裂法(如果要多个值可以返回数组)，展开/传递参数(使用数组参数结构)
    • 单决议：异步的事件/数据流
    • 惯性：（一个将普通基于回调的异步改为基于Promise的异步）Promise.wrap
    • 无法取消的Promise
