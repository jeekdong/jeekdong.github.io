简析JavaScript中的相等比较符

JavaScript提供三种不同的相等比较操作符:

  • 严格相等，使用 ==
  • (非严格)相等，使用 ===
  • 以及 Object.is （ECMAScript 6 新特性）

       众所周知，JavaScript中的等于操作符相对复杂，尤其是 == 操作符 ，运算规则奇怪，容易让人犯错，从而成为JavaScript中“最糟糕的特性”之一.想要弄清这几个操作符的区别，首先需要了解JavaScript的基本类型。

  • 5种简单数据类型:
    * Boolean
    * Null
    * Undefiend
    * Number
    * String

  • 1种复杂数据类型Object

  • Undefined类型和Null类型的都只有一个值，即undefined和null；Boolean类型有两个值：true和false；Number类型的值有很多很多；String类型的值理论上有无数个

三种相等比较符的定义

==:称作相等运算符（Equality Operator），它用来检测两个操作是否相等，这里的相等的定义非常宽松，可以允许类型的转换。


===:称作严格相等运算符(Strict Equality)，也被称之为恒等运算符(Identity Operator)或全等运算符，它用来检测两个操作数是否严格相等。


Object.is 确定两个值是否是相同的值。

全等符===

严格相等运算符===首先要计算其操作数的值，然后比较这两个值，比较过程没有任何类型转换。

  1. 如果两个被比较的值类型相同，值也相同，并且都不是 number 类型时，两个值全等。
  2. 如果两个值都是 number 类型，当两个都不是 NaN，并且数值相同，两个值全等。
  3. 或是两个值分别为 +0 和 -0 时，两个值被认为是全等的。

###非严格相等 == 非严格相等==操作符在运算时会进行类型转换，很容易让人迷糊. >==运算规则的精确描述：[The Abstract Equality Comparison Algorithm](http://es5.github.io/#x11.9.3)

网上看了很多资料发现@manxisuo提供的图片是最容易理解的。在此贴出 .

图中的两个符号：大写字母N和P。这两个符号的含义：

  • N表示ToNumber操作，即将操作数转为数字。它是规范中的抽象操作，但我们可以用JS中的*Number()*函数来等价替代。
  • P表示ToPrimitive操作，即将操作数转为原始类型的值。它也是规范中的抽象操作，同样也可以翻译成等价的JS代码。不过稍微复杂一些，简单说来，对于一个对象obj：

*ToPrimitive(obj)*等价于：先计算obj.valueOf()，如果结果为原始值，则返回此结果；否则，计算obj.toString()，如果结果是原始值，则返回此结果；否则，抛出异常。

非严格相等 ==运算规则详细图解

总结起来，具体为四条规则：

  • undefined == null，结果是true。且它俩与所有其他值比较的结果都是false。
  • String == Boolean，需要两个操作数同时转为Number。
  • String/Boolean == Number，需要String/Boolean转为Number。
  • Object == Primitive，需要Object转为Primitive(具体通过valueOf和toString方法)。

###Object.is()

Object.is 在三等号判等的基础上特别处理了 NaN 、 -0 和 +0 ，保证 -0 和 +0 不再相同，但 Object.is(NaN, NaN)会返回 true。



到底该使用哪一个相等比较符

先来看一个例子：


if (x == 10){
    x+=5；
}

       这串操作看起来没有任何问题，但是如果传入的x是字符串'10'，x的结果会变成'105'。在后续运算中字符串'105'又可能被转型，从而引入隐蔽的错误。

       从以上例子可以看出：== 的比较看似会比较方便，比如 1 == '1' ，但是会埋下隐患，比如可能对类型做出错误的假设。绝大多数场合应该使用 === ，只有检测 null/undefined 的时候可以使用 x == null ，因为通常我们不区分 null 和 undefined ，即将 x == null 作为 x === null || x === undefined 的缩写。

       Object.is唯一让人感兴趣的，是当你需要一些元编程方案时，它对待0的特殊方式，特别是关于属性描述器，即你的工作需要去镜像Object.defineProperty的一些特性时。如果你的工作不需要这些，那你应该避免使用Object.is
